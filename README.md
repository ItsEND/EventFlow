# EventFlow

Учебный проект ASP.NET Core Web API для управления мероприятиями и бронированиями.

Приложение реализует CRUD-операции для мероприятий, фильтрацию и пагинацию, создание бронирований с быстрым ответом `202 Accepted`, фоновую обработку бронирований, хранение данных в PostgreSQL, управление схемой через EF Core migrations, единый формат ошибок `ProblemDetails`, unit-тесты и интеграционные тесты репозиториев с Testcontainers.

## Что реализовано

- CRUD-операции для мероприятий;
- хранение мероприятий и бронирований в PostgreSQL;
- доступ к данным через паттерн Repository;
- управление схемой базы данных через EF Core migrations;
- автоматическое применение миграций при запуске приложения;
- бизнес-логика в сервисах;
- подключение сервисов и репозиториев через Dependency Injection;
- валидация входных данных;
- глобальная обработка исключений через middleware;
- единый формат ошибок через `ProblemDetails`;
- фильтрация мероприятий по названию и диапазону дат;
- пагинация списка мероприятий;
- создание брони через быстрый ответ `202 Accepted`;
- возврат заголовка `Location` при создании брони;
- получение текущего статуса брони;
- фоновая обработка бронирований через `BackgroundService`;
- очередь фоновых задач на основе `Channel<Guid>`;
- Swagger для ручной проверки API;
- unit-тесты бизнес-логики;
- интеграционные тесты репозиториев на реальном PostgreSQL через Testcontainers.

## Архитектура хранения данных

События и бронирования сохраняются в PostgreSQL. Работа с базой данных выполняется через EF Core и репозитории:

- `EventRepository` — получение, фильтрация, пагинация, добавление и удаление мероприятий;
- `BookingRepository` — добавление и получение бронирований;
- `AppDbContext` — EF Core-контекст приложения.

Схема базы данных управляется миграциями EF Core. Ручное выполнение SQL-скриптов для создания таблиц не требуется.

При запуске приложение вызывает `Database.Migrate()`: если база данных пустая, схема создаётся автоматически; если появились новые миграции, они применяются последовательно.

Очередь фоновой обработки бронирований хранится в памяти процесса приложения. Сами сущности `Event` и `Booking` сохраняются в PostgreSQL, но необработанные элементы очереди не являются персистентными и после перезапуска приложения должны быть поставлены в очередь повторно отдельным механизмом, если такой сценарий потребуется.

## Сущность Event

Событие содержит поля:

| Поле | Тип | Описание |
|---|---|---|
| `id` | `Guid` | Идентификатор события |
| `title` | `string` | Название события |
| `description` | `string?` | Описание события |
| `totalSeats` | `int` | Общее количество мест на мероприятии |
| `availableSeats` | `int` | Текущее количество свободных мест |
| `startAt` | `DateTime` | Дата и время начала |
| `endAt` | `DateTime` | Дата и время окончания |

## Сущность Booking

Бронь содержит поля:

| Поле | Тип | Описание |
|---|---|---|
| `id` | `Guid` | Идентификатор брони |
| `eventId` | `Guid` | Идентификатор мероприятия, к которому относится бронь |
| `status` | `string` | Текущий статус брони |
| `createdAt` | `DateTime` | Дата и время создания брони |
| `processedAt` | `DateTime?` | Дата и время обработки брони |

## Статусы брони

| Статус | Описание |
|---|---|
| `Pending` | Бронь создана и ожидает фоновой обработки |
| `Confirmed` | Бронь подтверждена |
| `Rejected` | Бронь отклонена |

В текущем сценарии фоновая обработка переводит бронь из `Pending` в `Confirmed`.

Статус `Rejected` используется, если бронь не может быть обработана успешно: например, если мероприятие было удалено к моменту фоновой обработки или при обработке произошла ошибка. При отклонении брони место может быть возвращено в пул доступных мест, если мероприятие ещё существует.

## Валидация

Для мероприятий проверяется:

- обязательность `title`;
- максимальная длина `description`;
- обязательность `totalSeats`;
- `totalSeats` должно быть больше `0`;
- обязательность `startAt`;
- обязательность `endAt`;
- `endAt` должно быть позже `startAt`.

Для пагинации проверяется:

- `page` не может быть меньше `1`;
- `pageSize` не может быть меньше `1`;
- `pageSize` не может быть больше `50`.

Для бронирований проверяется:

- нельзя создать бронь для несуществующего мероприятия;
- нельзя получить бронь по несуществующему идентификатору;
- нельзя повторно обработать уже обработанную бронь;
- нельзя создать бронь, если свободных мест нет.

## Фильтрация и пагинация мероприятий

Эндпоинт `GET /events` поддерживает query-параметры:

| Параметр | Описание |
|---|---|
| `title` | Поиск по названию: регистронезависимое частичное совпадение |
| `from` | Возвращает события, начинающиеся не раньше указанной даты |
| `to` | Возвращает события, заканчивающиеся не позже указанной даты |
| `page` | Номер страницы, по умолчанию `1` |
| `pageSize` | Количество элементов на странице, по умолчанию `10` |

Фильтрация, подсчёт общего количества записей, сортировка и пагинация выполняются на стороне PostgreSQL.

Пример:

```http
GET /events?title=митап&from=2026-05-01&to=2026-05-31&page=1&pageSize=5
```

## Как запустить проект

### 1. Клонировать репозиторий

```bash
git clone https://github.com/ItsEND/EventFlow.git
cd EventFlow
```

### 2. Настроить строку подключения

Приложение использует строку подключения `DefaultConnection`.

Пример для PowerShell:

```powershell
$env:ConnectionStrings__DefaultConnection = "Host=localhost;Port=5432;Database=eventflow;Username=postgres;Password=postgres"
```

Пример для Bash:

```bash
export ConnectionStrings__DefaultConnection="Host=localhost;Port=5432;Database=eventflow;Username=postgres;Password=postgres"
```

При необходимости локальный PostgreSQL можно запустить через Docker:

```bash
docker run --name eventflow-postgres \
  -e POSTGRES_DB=eventflow \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 \
  -d postgres:16-alpine
```

### 3. Восстановить зависимости

```bash
dotnet restore
```

### 4. Собрать solution

```bash
dotnet build
```

### 5. Запустить API

```bash
dotnet run --project EventFlow.Api
```

При запуске приложение автоматически применит EF Core migrations через `Database.Migrate()`.

## EF Core migrations

Схема PostgreSQL создаётся и обновляется миграциями EF Core. Ручной SQL для инициализации схемы не требуется.

Если инструмент `dotnet ef` ещё не установлен:

```bash
dotnet tool install --global dotnet-ef --version 10.0.9
```

Создать новую миграцию:

```bash
dotnet ef migrations add <MigrationName> \
  --project EventFlow.Api \
  --startup-project EventFlow.Api
```

Пример:

```bash
dotnet ef migrations add AddBookingIndexes \
  --project EventFlow.Api \
  --startup-project EventFlow.Api
```

Применить миграции вручную:

```bash
dotnet ef database update \
  --project EventFlow.Api \
  --startup-project EventFlow.Api
```

Обычно отдельный ручной вызов `database update` не требуется, поскольку приложение выполняет `Database.Migrate()` при запуске.

## Проверка через Swagger

Для финальной ручной проверки выполните следующий сценарий.

### 1. Создать мероприятие

```http
POST /events
Content-Type: application/json
```

```json
{
  "title": "Backend Conference",
  "description": "ASP.NET Core Web API practice",
  "totalSeats": 3,
  "startAt": "2026-07-01T10:00:00Z",
  "endAt": "2026-07-01T18:00:00Z"
}
```

Ожидаемый результат: `201 Created`. Сохраните `id` мероприятия из ответа.

### 2. Получить созданное мероприятие

```http
GET /events/{eventId}
```

Ожидаемый результат: `200 OK` и данные созданного мероприятия.

### 3. Создать бронь

```http
POST /events/{eventId}/book
```

Ожидаемый результат:

- `202 Accepted`;
- заголовок `Location: /bookings/{bookingId}`;
- статус брони `Pending` в теле ответа.

### 4. Получить бронь

```http
GET /bookings/{bookingId}
```

Сразу после создания ожидается статус `Pending`. После фоновой обработки повторный запрос должен вернуть статус `Confirmed` и заполненное поле `processedAt`.

## Эндпоинты API

### Мероприятия

| Метод | Маршрут | Описание |
|---|---|---|
| `GET` | `/events` | Получить список мероприятий с фильтрацией и пагинацией |
| `GET` | `/events/{id}` | Получить мероприятие по идентификатору |
| `POST` | `/events` | Создать мероприятие |
| `PUT` | `/events/{id}` | Обновить мероприятие |
| `DELETE` | `/events/{id}` | Удалить мероприятие |

### Бронирования

| Метод | Маршрут | Описание |
|---|---|---|
| `POST` | `/events/{id}/book` | Создать бронь для мероприятия |
| `GET` | `/bookings/{id}` | Получить текущее состояние брони |

## Фоновая обработка бронирований

Фоновая обработка реализована через `BackgroundService`.

Сценарий обработки:

1. клиент отправляет `POST /events/{id}/book`;
2. API проверяет наличие мероприятия и свободных мест;
3. создаётся бронь со статусом `Pending`;
4. API сразу возвращает `202 Accepted`;
5. идентификатор брони передаётся в очередь фоновой обработки;
6. фоновый сервис получает бронь из очереди;
7. выполняется искусственная задержка для имитации внешней системы;
8. бронь переводится в статус `Confirmed`;
9. поле `processedAt` заполняется датой и временем обработки.

Для фоновой обработки используется очередь на основе `Channel<Guid>`. Она позволяет быстро принять бронь через API, вернуть клиенту `202 Accepted`, а дальнейшую обработку выполнить в `BackgroundService`.

## Синхронизация и защита от овербукинга

При создании брони необходимо атомарно выполнить две операции:

1. проверить, что у мероприятия есть свободные места;
2. уменьшить количество свободных мест и сохранить бронь.

Если выполнить эти действия без синхронизации, параллельные запросы могут одновременно увидеть `availableSeats > 0` и создать больше броней, чем реально доступно мест.

Для защиты критической секции в `BookingService` используется `SemaphoreSlim`. Он подходит для сценария с асинхронными операциями внутри критической секции: обычный `lock` нельзя удерживать во время `await`.

## Формат ошибок

Для ошибок используется единый JSON-формат `ProblemDetails`.

### Пример ошибки валидации

```json
{
  "type": "https://httpstatuses.com/400",
  "title": "Некорректный запрос",
  "status": 400,
  "detail": "Одна или несколько ошибок валидации",
  "instance": "/events"
}
```

### Пример ошибки, если ресурс не найден

```json
{
  "type": "https://httpstatuses.com/404",
  "title": "Ресурс не найден",
  "status": 404,
  "detail": "Event с ключом 2f2f8d3e-4ef5-4f8e-bd95-2fcd11a54c6b не найден.",
  "instance": "/events/2f2f8d3e-4ef5-4f8e-bd95-2fcd11a54c6b"
}
```

### Пример ошибки отсутствия свободных мест

```json
{
  "type": "https://httpstatuses.com/409",
  "title": "Конфликт",
  "status": 409,
  "detail": "Свободных мест на это мероприятие нет.",
  "instance": "/events/e1b0f2a7-1bb7-45f5-a54f-08a70cbe64c4/book"
}
```

## HTTP-статусы

| Статус | Описание |
|---|---|
| `200 OK` | Успешное получение или обновление ресурса |
| `201 Created` | Успешное создание мероприятия |
| `202 Accepted` | Бронь создана и принята в фоновую обработку |
| `204 No Content` | Успешное удаление мероприятия |
| `400 Bad Request` | Ошибка валидации |
| `404 Not Found` | Ресурс не найден |
| `409 Conflict` | Нет свободных мест или возник конфликт операции |
| `500 Internal Server Error` | Непредвиденная ошибка сервера |

## Тесты

### Unit-тесты

Unit-тесты проверяют бизнес-логику доменных сущностей и сервисов, в том числе:

- создание, получение, изменение и удаление мероприятий;
- фильтрацию и пагинацию мероприятий;
- комбинированные фильтры;
- создание и получение бронирований;
- изменение статусов бронирования;
- сценарии с некорректными входными данными;
- сценарии с отсутствующими ресурсами.

### Интеграционные тесты репозиториев

Интеграционные тесты проверяют работу `EventRepository` и `BookingRepository` на реальном PostgreSQL.

Для тестов используется Testcontainers:

- PostgreSQL автоматически запускается в Docker-контейнере;
- строка подключения получается из объекта контейнера;
- порт PostgreSQL не захардкожен;
- все интеграционные тесты используют один контейнер;
- перед каждым тестом база приводится к чистому состоянию;
- схема создаётся через `EnsureDeletedAsync()` и `MigrateAsync()`;
- тесты не зависят от порядка запуска.

Для запуска интеграционных тестов Docker daemon должен быть запущен.

Запустить все тесты:

```bash
dotnet test
```

Запустить только интеграционные тесты:

```bash
dotnet test EventApi.IntegrationTests
```

Повторный запуск `dotnet test` должен завершаться стабильно: контейнер поднимается автоматически, а состояние БД очищается между тестами.
