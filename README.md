# EventFlow

Учебный проект ASP.NET Core Web API для управления мероприятиями и бронированиями.

Проект реализует CRUD-операции для мероприятий, фильтрацию, пагинацию, глобальную обработку ошибок, unit-тесты, а также паттерн быстрого ответа с отложенной фоновой обработкой бронирований.

## Что реализовано

- CRUD-операции для мероприятий
- хранение данных в памяти приложения
- бизнес-логика вынесена в сервисы
- сервисы подключены через Dependency Injection
- валидация входных данных
- глобальная обработка исключений через middleware
- единый формат ошибок через ProblemDetails
- фильтрация мероприятий по названию и диапазону дат
- пагинация списка мероприятий
- сущность Booking для бронирования мероприятий
- создание брони через быстрый ответ `202 Accepted`
- возврат заголовка `Location` при создании брони
- получение текущего статуса брони
- фоновая обработка бронирований через `BackgroundService`
- искусственная задержка обработки брони для имитации внешней системы
- Swagger для тестирования API
- unit-тесты для `EventService`, `BookingService`, `Booking`

## Сущность Event

Событие содержит поля:

| Поле | Тип | Описание |
|---|---|---|
| `id` | `Guid` | Идентификатор события |
| `title` | `string` | Название события |
| `description` | `string?` | Описание события |
| `totalSeats` | `int` | Общее количество мест на мероприятии |
| `availableSeats` | `int` | Текущее количество свободных мест |
| `startAt` | `DateTime` | Дата и время начала |
| `endAt` | `DateTime` | Дата и время окончания |

## Сущность Booking

Бронь содержит поля:

| Поле | Тип | Описание |
|---|---|---|
| `id` | `Guid` | Идентификатор брони |
| `eventId` | `Guid` | Идентификатор мероприятия, к которому относится бронь |
| `status` | `string` | Текущий статус брони |
| `createdAt` | `DateTime` | Дата и время создания брони |
| `processedAt` | `DateTime?` | Дата и время обработки брони |

## Статусы брони

| Статус | Описание |
|---|---|
| `Pending` | Бронь создана и ожидает фоновой обработки |
| `Confirmed` | Бронь подтверждена |
| `Rejected` | Бронь отклонена |

В текущей реализации фоновая обработка переводит бронь из `Pending` в `Confirmed`.

Статус `Rejected` используется, если бронь не может быть обработана успешно: например, если мероприятие было удалено к моменту фоновой обработки или при обработке произошла ошибка. При отклонении брони место может быть возвращено в пул доступных мест, если мероприятие ещё существует.

## Валидация

Для мероприятий проверяется:

- обязательность `title`
- максимальная длина `description`
- обязательность `totalSeats`
- `totalSeats` должно быть больше `0`
- обязательность `startAt`
- обязательность `endAt`
- `endAt` должно быть позже `startAt`

Для пагинации проверяется:

- `page` не может быть меньше `1`
- `pageSize` не может быть меньше `1`
- `pageSize` не может быть больше `50`

Для бронирований проверяется:

- нельзя создать бронь для несуществующего мероприятия
- нельзя создать бронь для удалённого мероприятия
- нельзя получить бронь по несуществующему идентификатору
- нельзя повторно обработать уже обработанную бронь

## Фильтрация и пагинация мероприятий

Эндпоинт `GET /events` поддерживает query-параметры:

| Параметр | Описание |
|---|---|
| `title` | Поиск по названию, регистронезависимое частичное совпадение |
| `from` | Возвращает события, начинающиеся не раньше указанной даты |
| `to` | Возвращает события, заканчивающиеся не позже указанной даты |
| `page` | Номер страницы, по умолчанию `1` |
| `pageSize` | Количество элементов на странице, по умолчанию `10` |

Пример:

```http
GET /events?title=митап&from=2026-05-01&to=2026-05-31&page=1&pageSize=5
```

## Бронирования

### Создать бронь для мероприятия

```http
POST /events/{id}/book
```

Создаёт бронь для существующего мероприятия.

Особенности:

- API возвращает `202 Accepted`, потому что обработка выполняется асинхронно
- в заголовке `Location` возвращается ссылка на созданную бронь
- бронь создаётся в статусе `Pending`
- фоновый сервис позднее переводит бронь в статус `Confirmed`
- если мероприятие не найдено, возвращается `404 Not Found`

Пример успешного ответа:

```json
{
  "id": "4c93f90e-4fd8-4b2d-8a42-765f9f1de1f7",
  "eventId": "e1b0f2a7-1bb7-45f5-a54f-08a70cbe64c4",
  "status": "Pending",
  "createdAt": "2026-04-28T12:00:00Z",
  "processedAt": null
}
```

Пример заголовка ответа:

```http
Location: /bookings/4c93f90e-4fd8-4b2d-8a42-765f9f1de1f7
```

Возможные ответы:

- `202 Accepted` — бронь создана и ожидает фоновой обработки
- `404 Not Found` — мероприятие не найдено
- `409 Conflict` — свободных мест на мероприятие больше нет
### Получить бронь по id

```http
GET /bookings/{id}
```

Возвращает текущее состояние брони.

Возможные ответы:

- `200 OK` — бронь найдена
- `404 Not Found` — бронь не найдена
- `409 Conflict` — конфлик
Пример ответа сразу после создания брони:

```json
{
  "id": "4c93f90e-4fd8-4b2d-8a42-765f9f1de1f7",
  "eventId": "e1b0f2a7-1bb7-45f5-a54f-08a70cbe64c4",
  "status": "Pending",
  "createdAt": "2026-04-28T12:00:00Z",
  "processedAt": null
}
```

Пример ответа после фоновой обработки:

```json
{
  "id": "4c93f90e-4fd8-4b2d-8a42-765f9f1de1f7",
  "eventId": "e1b0f2a7-1bb7-45f5-a54f-08a70cbe64c4",
  "status": "Confirmed",
  "createdAt": "2026-04-28T12:00:00Z",
  "processedAt": "2026-04-28T12:00:02Z"
}
```

Бронирование невозможно из-за отсутствия свободных мест

```json
{
  "type": "https://httpstatuses.com/409",
  "title": "Конфликт",
  "status": 409,
  "detail": "Свободных мест на это мероприятие нет.",
  "instance": "/events/e1b0f2a7-1bb7-45f5-a54f-08a70cbe64c4/book"
}
```

## Фоновая обработка бронирований

Фоновая обработка реализована через `BackgroundService`.

Сценарий обработки:

1. клиент отправляет запрос `POST /events/{id}/book`
2. API проверяет, что мероприятие существует
3. создаётся бронь в статусе `Pending`
4. API сразу возвращает `202 Accepted`
5. идентификатор брони передаётся в очередь фоновой обработки
6. фоновый сервис получает бронь из очереди
7. выполняется искусственная задержка для имитации внешней системы
8. бронь переводится в статус `Confirmed`
9. поле `processedAt` заполняется датой и временем обработки

В текущей реализации данные о бронированиях хранятся в памяти приложения.

## Синхронизация и защита от овербукинга

При создании брони необходимо атомарно выполнить две операции:

1. проверить, что у мероприятия есть свободные места;
2. уменьшить количество свободных мест и сохранить бронь.

Если эти действия выполнить без синхронизации, несколько параллельных запросов могут одновременно увидеть `availableSeats > 0` и создать больше броней, чем реально доступно мест.

Для защиты критической секции в `BookingService` используется `SemaphoreSlim`.

Он защищает операции:

- получение мероприятия;
- проверку доступных мест через `TryReserveSeats()`;
- уменьшение `AvailableSeats`;
- создание и сохранение брони.

Используется именно `SemaphoreSlim`, а не `lock`, потому что внутри критической секции есть асинхронные операции с `await`. Обычный `lock` нельзя использовать вместе с `await`.

Для фоновой обработки бронирований используется очередь на основе `Channel<Guid>`. Она позволяет быстро принять бронь через API, вернуть клиенту `202 Accepted`, а дальнейшую обработку выполнить в `BackgroundService`.

Фоновый сервис обрабатывает брони параллельно через `Parallel.ForEachAsync` с ограничением степени параллелизма.


## HTTP-статусы

| Статус | Описание |
|---|---|
| `200 OK` | Успешное получение или обновление ресурса |
| `201 Created` | Успешное создание мероприятия |
| `202 Accepted` | Бронь создана и принята в фоновую обработку |
| `204 No Content` | Успешное удаление мероприятия |
| `400 Bad Request` | Ошибка валидации |
| `404 Not Found` | Ресурс не найден |
| `500 Internal Server Error` | Непредвиденная ошибка сервера |

## Особенности реализации

Данные хранятся в памяти приложения.

Для мероприятий используется in-memory-хранилище на основе коллекции событий.

Для бронирований используется in-memory-хранилище бронирований и очередь фоновой обработки.

После перезапуска приложения все созданные мероприятия и бронирования удаляются.
